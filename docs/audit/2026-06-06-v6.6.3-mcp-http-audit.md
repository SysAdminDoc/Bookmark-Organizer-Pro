# v6.6.3 MCP Streamable HTTP Audit

Date: 2026-06-06

## Research Findings

- MCP Streamable HTTP carries the same protocol semantics as stdio, but mirrors selected request metadata into HTTP headers for routing and observability.
- SEP-2243 requires `Mcp-Method` on requests/notifications and `Mcp-Name` for named operations such as `tools/call`, `resources/read`, and `prompts/get`.
- FastMCP 3.4.2 exposes HTTP transport through `run(transport="http", host=..., port=..., path=...)` and accepts `stateless_http=True`.
- FastMCP 3.4.2 does not currently enforce `Mcp-Method`/`Mcp-Name` header matching itself, so BOP needs a local middleware guard.

## Code Findings

- The existing CLI only exposed MCP over stdio.
- The MCP server already had a reusable FastMCP builder, so HTTP transport could share the same tool catalog, cache hints, annotations, auth behavior, and stateless metadata.
- Header validation needs to consume the POST body before FastMCP, then replay the same body to the downstream ASGI app.

## Changes Shipped

- Added `serve_http(host, port, path)` to run the FastMCP Streamable HTTP server.
- Added `mcp-http-server [--host HOST] [--port N] [--path /mcp]` to the CLI.
- Kept loopback binding as the default host.
- Enabled `stateless_http=True` for the HTTP transport.
- Added `MCPHTTPHeaderValidationMiddleware` to reject mismatched `Mcp-Method` and `Mcp-Name` headers with HTTP 400.
- Bumped release metadata to v6.6.3.
