# v6.6.2 MCP Stateless Readiness Audit

Date: 2026-06-06

## Research Findings

- MCP stateless work is now a final SEP, with per-request protocol metadata and `server/discover` replacing connection-coupled assumptions.
- The draft transport docs describe identical protocol semantics across stdio and Streamable HTTP, with HTTP only mirroring selected metadata into headers.
- The draft caching docs define `ttlMs` and `cacheScope` as cache hints for list results.
- The installed MCP SDK supports a `stateless` run flag and allows extra fields on `ListToolsResult`.
- FastMCP 3.4.2 supports `stateless=True` on stdio and accepts tool `annotations` plus tool `meta`.

## Code Findings

- BOP currently exposes MCP over stdio, not Streamable HTTP, so `Mcp-Method` and `Mcp-Name` header validation cannot be exercised without adding an HTTP transport.
- Raw SDK `tools/list` could return a typed `ListToolsResult`, which is the right boundary for cache hints.
- FastMCP's default `tools/list` handler omitted result-level cache hints, but the low-level handler can be wrapped without changing tool implementations.
- Existing pure tool implementations were reusable; the needed work was protocol metadata and transport configuration.

## Changes Shipped

- Enabled stateless mode for raw SDK stdio and FastMCP stdio server runs.
- Added `ttlMs` and `cacheScope` to raw SDK and FastMCP `tools/list` results.
- Added read-only, destructive, idempotent, open-world, and stateless metadata to raw SDK and FastMCP tool catalogs.
- Kept the existing tool tuple shape intact and centralized MCP metadata in helper functions.
- Updated MCP server module docs to stay client-neutral.
- Bumped release metadata to v6.6.2.
