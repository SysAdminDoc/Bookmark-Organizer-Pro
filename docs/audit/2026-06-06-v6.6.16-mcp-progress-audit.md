# v6.6.16 MCP Progress Audit

Date: 2026-06-06
Scope: R-15 partial, FastMCP progress notifications for RAG chat streaming

## Objective

Connect the stream-shaped RAG chat result to MCP progress notifications without
changing the final tool payload. Clients that supply an MCP progress token can
now receive progress updates for ordered response chunks; clients that do not
consume progress still receive the same JSON result.

## Changes

- Added `_report_chat_stream_progress()` to replay chunk events through
  `ctx.report_progress()`.
- Updated the FastMCP `chat_with_collection_stream` registration to accept the
  injected context when FastMCP exposes `Context`.
- Preserved the existing raw SDK and no-context FastMCP fallback paths.
- Kept `ctx` out of the public tool schema by relying on FastMCP context
  injection.
- Added MCP tests for progress calls and schema hygiene.

## Compatibility

- `chat_with_collection_stream` still returns the same `answer`, `mode`,
  `provider_streaming`, `events`, source, and provenance fields.
- Clients without progress tokens receive no extra transport behavior.
- Raw MCP tool calls continue to return the event payload without FastMCP
  progress replay.

## Remaining R-15 Work

- Build an end-to-end local MCP client smoke that asserts progress
  notifications over stdio or Streamable HTTP.
- Evaluate true transport-level token delivery if FastMCP exposes a better
  primitive than progress notification replay.
- Decide whether Anthropic and Google native streaming adapters should be added
  after SDK surface verification.
